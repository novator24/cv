# Архитектура системы: Трекинг и сортировка товаров на складе (VLM + YOLO)

Данная архитектура адаптирует подход использования легковесных Vision-Language Models (таких как Liquid AI LFM2-VL-3B) и YOLOv8n для решения складской задачи: **мониторинга и сортировки товаров при синхронизированной съемке с нескольких камер**.

## Описание задачи
На современных складах процесс ручной или полуавтоматической сортировки подвержен человеческому фактору (товар положили не в ту корзину, пропустили этап сканирования). Обычные системы компьютерного зрения (CV) могут находить коробки в кадре, но им не хватает **контекстного понимания** действий (например, "Работник взял коробку с ленты, осмотрел её и положил в синюю зону отгрузки").

Использование 3B VLM с темпоральной памятью (анализ последовательности кадров) позволяет решить эту задачу в реальном времени на доступном оборудовании (Edge-серверы с потребительскими или начальными промышленными GPU).

## Компоненты системы

Архитектура состоит из нескольких контейнеризированных микросервисов, разворачиваемых в Kubernetes кластере:

### 1. Подсистема захвата и синхронизации (Ingress Node)
* **Мультикамерный стриминг**: Прием RTSP-потоков с нескольких камер, установленных под разными углами над зоной сортировки.
* **Синхронизация по времени (Timestamping)**: Кадры выравниваются по времени, формируя единый "срез" (например, 15 FPS).
* **Frame Skipper**: Прореживание кадров (анализ каждого 2-го или 3-го кадра), так как складские действия занимают несколько секунд. Это снижает нагрузку на сеть и GPU.

### 2. Подсистема детекции объектов (Object Detection Node - YOLOv8n)
* Изолированная легковесная модель YOLOv8n находит на кадрах базовые объекты: `работник`, `посылка/коробка`, `штрихкод`, `сортировочная корзина`.
* Отправляет координаты bounding boxes в следующий модуль.
* **Цель**: Защита от "галлюцинаций" VLM. VLM лучше понимает сцену, если ему передать точные координаты ключевых объектов.

### 3. Темпоральная память (Temporal Memory Buffer)
* Буферизирует кадры для передачи в VLM. Оптимальное окно: **N=3 кадра** (прошлое, недавнее прошлое, текущий момент) со всех синхронизированных ракурсов.

### 4. VLM Анализатор (LFM2-VL-3B Node)
* **Вход**: 3 последовательных кадра с каждой из камер + JSON с bounding boxes от YOLO + структурированный системный промпт (например: "Опиши действия работника с посылкой").
* **Обработка**: 3-миллиардная модель (Liquid AI LFM2-VL-3B или аналогичная, например, Qwen-VL) анализирует взаимосвязи: откуда взяли товар и куда его поместили.
* **Выход**: Текстовое описание или JSON (например: `{"action": "sort", "item": "box_1", "destination": "bin_A", "confidence": 0.95}`).

### 5. Event Extraction & Integration
* Модуль, который парсит ответы VLM, фильтрует их с низким `temperature` (чтобы результаты были детерминированы) и отправляет события в Warehouse Management System (WMS) или в Kafka для дальнейшей аналитики.

## Почему это работает лучше традиционного CV?
1. **Понимание намерений**: Традиционный CV скажет: "Коробка и руки работника пересекаются". VLM с темпоральной памятью скажет: "Работник перекладывает коробку из зоны А в зону Б".
2. **Стоимость**: Использование легковесной модели (3B параметров) позволяет запускать инференс локально (Edge AI) без затрат на API облачных гигантов (GPT-4V), что критично при потоковом видео (24/7).
3. **Устойчивость к окклюзии**: Благодаря съемке с разных ракурсов и синхронизации, если на одной камере коробка перекрыта спиной работника, VLM может использовать контекст с другой камеры.
